Your talking guide for the Beff project

Short bullets to glance at. A short line you can read out loud if you freeze. Keep this open on your phone or a second tab.

1The 30-second pitch

Start with this when they say "tell me about your project."

  • Beff is a pre-launch B2B website for a green-tech company.
  • The task: build a landing page, sign-up, login, dashboard, and profile.
  • The profile must include two GDPR features — I built data export and account delete.
  • I built the frontend with React and the backend with NestJS.
  • It is live on my server at bff.i-ash.com, with real user sign-up and login.
I built a small SaaS web app for a green-tech startup that has not launched yet. It has a landing page, sign-up, login, a dashboard, and a profile page with two privacy features. The frontend is React, the backend is NestJS, and it is live on my own server.

2The frontend

If they ask "what did you use?" — keep the words simple.

  • React with TypeScript for the pages.
  • Tailwind CSS for styling — it lets me design fast.
  • Vite as the build tool. It is faster than older tools.
  • I used shadcn/ui — small, ready-made components I copy into my code.
  • The forms use react-hook-form + zod. Zod checks the input on the screen.
The frontend is React with TypeScript, styled with Tailwind. I used a small set of ready-made components called shadcn so the design feels clean. Forms are validated on both sides — in the browser and on the server.

3The backend

In plain words. Skip the deep tech unless they push.

  • NestJS — a Node.js framework that keeps the code tidy.
  • It has two main parts: one for sign-up and login, one for the user profile.
  • Passwords are hashed with bcrypt — never saved as plain text.
  • After login the server gives back a token (JWT). The browser sends it on every request.
  • Users are saved in a small file database called SQLite. It survives restarts.
The backend is NestJS. When a user signs up, I hash the password with bcrypt and save the user in a small SQLite database. Login gives back a token. The browser sends that token on every request to prove who the user is.

4The two GDPR features

This is what the task asked for. Be ready to demo.

  • Download my data — the user gets a JSON file with their info. GET /users/me/export
  • Delete my account — removes the user. A pop-up asks them to confirm first. DELETE /users/me
  • Both pages live under /privacy in the app.
  • After delete, the old token stops working right away — the server checks the user still exists.
  • Both features show a toast message so the user knows what happened.
GDPR says people must be able to take their data out and delete their account. I built both. Download gives a JSON file. Delete asks for a confirm first, removes the user, and the old token stops working at once.

5Testing — the QA part

This is your big section. Take your time here.

Backend tests

  • Unit tests — I test one piece of code on its own. Fast and small.
  • End-to-end tests — I start the real server and call it like a real user. Slower, but real.
  • I cover register, login, get profile, edit name, export, delete.
  • I test the error paths too — wrong password, missing token, deleted user.

Frontend tests

  • Component tests with Vitest — I check that each page shows what it should.
  • Browser tests with Playwright — a real browser clicks through sign-up, login, profile.
  • I also test the redirects — a logged-out user sent to login, a logged-in user sent to dashboard.

Accessibility tests

  • I run axe-core on every page in the browser tests.
  • It checks color contrast, missing labels, broken landmarks.
  • Result: zero violations on all five pages.

The "restart" test (the one I'm proud of)

  • Register a user.
  • Close the app.
  • Start the app again on the same database file.
  • Try to log in with the same email and password — it works.
  • This proves the data really survives a server restart.

Why these tests, not more

  • I test the risky paths — auth, data, redirects.
  • I do not chase 100% line coverage. That is wasted time on simple code.
  • If a bug would cost the user trust, I write a test for it.
On the backend I have unit tests and end-to-end tests. On the frontend I have component tests and Playwright browser tests, plus an accessibility check on every page. My favourite test is the restart test: it signs up a user, closes the app, opens it again on the same database, and logs back in. It proves the storage really works.

6Hard parts I solved

Pick one or two of these if they ask "what was the hardest part?"

  • Production was calling localhost. The browser kept asking for localhost:3001 on the live site. I fixed the build setup so the live site uses the right URL.
  • Every deploy was wiping all signups. The server stored users in memory, so a restart erased them. I switched to SQLite — a small file that lives on the server and is not touched by deploys.
  • Four tests broke after I moved a feature. Old tests still pointed at the old page. Instead of patching them, I split them into two clean test files — one for each page.
The hardest part was that every deploy was wiping the signups. I had used a memory store. I moved to SQLite — a small file database — and added a test that proves the data survives a restart. That problem could not happen again.

7What I would do next

Shows you know what's missing without sounding negative.

  • Set up GitHub Actions to run all tests on every push.
  • Add a coverage report so I can see exactly what is and isn't tested.
  • Add rate limiting on login — so no one can guess passwords too fast.
  • Write more tests for the Dashboard page (it uses mock data right now).
  • Move from SQLite to Postgres if the user count grows.
The next steps are clear: a CI pipeline so tests run on every push, a coverage report, and rate limiting on login. If the project grew, I would move from SQLite to Postgres. I know where the gaps are.

8Quick answers to likely questions

Glance here if a question catches you off guard.

"Why JWT in one sentence?"

A JWT is a small signed token that proves who you are, so the server does not have to remember every login.

"Why SQLite, not Postgres?"

The task said keep it simple. SQLite is one file on the server. No setup, no extra service, and it survives restarts. I would switch to Postgres for real scale.

"Why no Passport for auth?"

Passport adds five packages for the same outcome. I wrote one small guard that checks the token. Less code, same security.

"How do you stop someone guessing emails?"

Wrong password and unknown email both return the same error. So an attacker can't tell which emails are real.

"What did you mock in tests, and why?"

In end-to-end tests I mock nothing — I want it real. In unit tests I use an in-memory database so they run fast. The risky paths are always tested against real code.

"How would you scale this?"

Move the database to Postgres on a separate server. Run more app servers behind a load balancer. Add a cache for read-heavy pages.

"How is the password stored?"

Hashed with bcrypt. Never plain text. The hash never leaves the user service — even the export feature strips it out.
