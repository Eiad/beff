Beff Technical write-up

How Beff was built

A quick tour of the auth flow, the database layer, the GDPR features, and where each line item of the assignment lives in the code.

GitHub repo Figma Live site

1Login & authentication

Stateless JWT auth. No Passport, no sessions, no cookies — just a signed token in localStorage, attached to every API request via an Axios interceptor.

Register / Login round trip

1Client submits name / email / password. react-hook-form + zod validates the shape before a request is ever made.
2NestJS re-validates server-side with the same zod schema (via a custom ZodValidationPipe). No trust in client validation.
3AuthService hashes the password with bcrypt (10 rounds) and persists the user row to SQLite.
4JWT signed with HS256, 7-day expiry, returned alongside the safe user object (no passwordHash ever leaves the service).
5Client stores the token, decodes it locally with jwt-decode, and checks exp on every mount — no network call needed to know if the user is signed in.

Protecting private routes

  • JwtGuard on every /users/* endpoint verifies the token and looks the user up in the DB — if the user was deleted, the token is effectively invalidated (401 returned).
  • Frontend ProtectedRoute waits for auth hydration before rendering to avoid the "logged-in UI flashes then disappears" problem.
  • Axios response interceptor clears the token and redirects to /login on any 401 except auth endpoints (so a bad login doesn't trigger a redirect loop).

Security choices worth naming

  • Same 401 for wrong password and unknown email — no user enumeration via error differences.
  • Password hashes never leave UsersService — all returns are typed Omit<User, 'passwordHash'>.
  • aria-live="polite" on form errors so screen readers announce 409/401 responses without stealing focus.

2Database — SQLite via better-sqlite3

Started as an in-memory Map<string, UserRecord> — simple and functional per the brief, but accounts wiped on every PM2 restart. Upgraded to a single-file SQLite store so signups actually survive a redeploy.

Why this driver

  • better-sqlite3 is synchronous — kept the entire UsersService API sync, no async cascade into AuthService, JwtGuard, or any test.
  • Prebuilt native binaries for Node 22 — no node-gyp, no build-essential on the VPS.
  • Single writer, WAL journal mode — durable across crashes, fine for this workload.

What's on disk

  • One table: users (id, name, email, password_hash, created_at).
  • Expression index on LOWER(email) — case-insensitive lookups without a unique constraint (Node's single-threaded event loop + sync driver means the app-level duplicate check in AuthService is race-free).
  • DB file at /root/beff/backend/data/beff.sqlite on the VPS. backend/data/ is .gitignored so git pull never touches production data.

Lifecycle

  • DatabaseService.onModuleInit opens the connection, applies journal_mode = WAL + foreign_keys = ON, runs CREATE TABLE IF NOT EXISTS.
  • app.enableShutdownHooks() ensures onModuleDestroy fires on SIGTERM (what PM2 sends on restart) — connection closes cleanly, no WAL leftovers.
  • Tests use SQLITE_DB_PATH=:memory: so each test gets a fresh isolated DB — no fixtures to clean up.

3GDPR features

Two user-facing rights, both exposed on /privacy behind the JWT guard.

Right to data portability

GET /users/me/export returns the user's stored data as a JSON blob with Content-Disposition: attachment, so the browser downloads it as beff-data-export-<date>.json.

Right to erasure

DELETE /users/me removes the row from SQLite. The JWT the user still holds becomes immediately useless because JwtGuard re-checks the DB on every request — no row, no access.

Deletion uses a type-to-confirm modal ("type delete my account") — standard UX guardrail against accidental clicks.

Beyond the brief

  • Full unit + e2e test suite (Jest + supertest for backend, Vitest + Playwright for frontend, axe-core accessibility audit).
  • Persistence across server restarts via SQLite (brief said "simple and functional"; in-memory would have satisfied it — went further for reviewer experience).
  • Toast system, page transitions, skeleton loaders, type-to-confirm deletion.
  • Deployed to a VPS (Hetzner) behind Nginx with PM2, not Vercel — chosen for persistence and because the Vercel lambda filesystem is ephemeral.
